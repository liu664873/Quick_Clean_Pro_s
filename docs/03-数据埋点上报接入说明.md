# QuickCleanPRO 数据埋点上报接入说明

本文档按产品当前口径和 `D:\pros\AdvertiseExample` 的 SDK 接入方式整理 QuickCleanPRO 的埋点边界。宿主只补业务事件和少量用户属性，SDK 已经自动完成的数据不重复上报。

## 1. 接入原则

1. 宿主 App 不直接接 ThinkingData、Firebase、Singular，统一通过广告 SDK 暴露的 `AdvertiseSdk` 上报。
2. 业务事件统一走 `AnalyticsTracker.track(...)`，内部调用 `AdvertiseSdk.logEvent(...)`。
3. 公共事件属性统一走 `AnalyticsTracker.setSuperProperties(...)`，内部调用 `AdvertiseSdk.setSuperProperties(...)`。
4. 用户属性只能在产品允许的白名单内写入；不在白名单内的字段，即使历史产品表出现过，也不在宿主上传。
5. SDK 内部已经自动上报的事件或用户属性，宿主不补报、不二次写入。
6. 广告生命周期事件由广告 SDK 根据 `areaKey` 自动上报，宿主只负责传正确广告位 key。

### 1.1 用户属性白名单

| 字段 | 写入方 | 说明 |
| --- | --- | --- |
| `device_id` | 宿主 | `AdvertiseSdk.getThinkingDeviceId()`，只写一次。 |
| `first_open_time` | 宿主 | `AdvertiseSdk.getFirstOpenTime()`，只写一次。 |
| `latest_open_time` | 宿主 | 初始化和进入前台时刷新。 |
| `total_open_num` | 宿主 | `AnalyticsTracker.initialize()` 中每进程初始化累计一次。 |
| `total_use_time` | 宿主 | `ProcessLifecycleOwner` 进入后台时累计并上传，单位毫秒。当前 SDK 无公开 `userAdd` API，宿主采用本地累计后退出时覆盖总值。 |
| `has_notification_permission` | SDK | SDK 生命周期监听中刷新，宿主不重复写。 |
| `has_allfile_permission` | SDK | SDK 生命周期监听中刷新，宿主不重复写。 |
| `ab_test` | SDK | Remote Config 的 `ABTestName` 写入。 |
| `network` | SDK | Singular 归因回调写入。 |

宿主不再上传这些历史字段：`total_cleancpl_num`、`total_battery_num`、`total_largefile_num`、`ad_strategy`、`singular_media_source`、`singular_campaign`、`singular_adset`、`singular_ad`。其中清理完成次数仅作为本地状态保留，用于通知权限弹窗 `noticeflag=3` 的判断。

### 1.2 SDK 自动上报不重复

| 类型 | SDK 自动内容 | 宿主要求 |
| --- | --- | --- |
| 广告生命周期 | `ad_occur`、`ad_start_loading`、`ad_finish_loading`、`ad_impression`、`ad_click`、`ad_close`、广告收入事件 | 不手动调用 `AnalyticsTracker.track(...)` 或 `AdvertiseSdk.logEvent(...)` 补报这些事件。 |
| App 自动采集 | `ta_app_install`、`ta_app_start`、`ta_app_view`、`ta_app_end`、`ta_app_crash` | 按 SDK/产品口径视为 SDK 内部完成，宿主不补报。 |
| 用户属性 | `has_notification_permission`、`has_allfile_permission`、`ab_test`、`network` | 宿主不二次调用 `setUserAttr` 写入。 |

## 2. 统一埋点入口

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/core/analytics/AnalyticsTracker.kt`

职责：

1. 集中维护宿主业务事件名、属性名、固定取值。
2. 统一过滤空参数，避免上报空字符串或 `null` 参数。
3. 封装业务事件上报：`track(eventName, params)`。
4. 封装公共属性上报：`setSuperProperties(properties)`。
5. 只写宿主负责的用户属性：`device_id`、`first_open_time`、`latest_open_time`、`total_open_num`；`total_use_time` 只在退出/进后台时累计上传。
6. 负责功能 `FeatureKey` 到产品表 `details` 的映射。
7. 负责启动来源 `AppOpenFrom` 到 `traffic_source` 的映射。

相关边界：

| 职责 | 代码位置 |
| --- | --- |
| 业务埋点和宿主用户属性统一入口 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/analytics/AnalyticsTracker.kt` |
| 功能操作事件转业务埋点/本地状态 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/common/operation/FeatureOperationTracker.kt` |
| 功能页离开语义封装 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/common/operation/FeatureExitHandler.kt` |
| 文件权限弹窗/授权业务事件 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/skin/permission/PermissionAnalytics.kt` |
| 权限请求流程和结构化结果 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/skin/permission/PermissionCoordinatorState.kt` |

常用方法：

```kotlin
AnalyticsTracker.initialize()
AnalyticsTracker.onAppForeground()
AnalyticsTracker.onAppBackground()
AnalyticsTracker.track(eventName, params)
AnalyticsTracker.handleLaunchIntent(intent)
AnalyticsTracker.trackCoreFeatureClicked(feature)
AnalyticsTracker.trackCoreFeatureEntered(feature)
AnalyticsTracker.trackFeatureOperation(event)
AnalyticsTracker.trackNotificationPermissionResult(accepted)
AnalyticsTracker.trackFilePermissionResult(accepted)
```

## 3. App 启动和用户属性

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/app/MyApp.kt`

接入方式：

1. `AdvertiseSdk.init(...)` 成功后调用 `AnalyticsTracker.initialize()`。
2. 初始化时写入宿主负责的启动类用户属性：
   - `device_id`
   - `first_open_time`
   - `latest_open_time`
   - `total_open_num`
3. 通过 `ProcessLifecycleOwner` 监听 App 前后台：
   - 前台：记录本次使用开始时间，并刷新 `latest_open_time`。
   - 后台：计算本次前台停留时长，累加到本地 `analytics_total_use_time`，再用 `setUserAttr(total_use_time, 累计总值)` 上传。

说明：`total_use_time` 不在启动初始化时上传；`MainActivity` 不再调用 `AnalyticsTracker.onAppForeground()` / `AnalyticsTracker.onAppBackground()`，避免 Activity 生命周期和进程生命周期重复累计使用时长。

## 4. 启动来源和通知点击

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/app/MainActivity.kt`

接入方式：

1. `onCreate(intent)` 调用 `AnalyticsTracker.handleLaunchIntent(intent)`。
2. `onNewIntent(intent)` 再次调用 `AnalyticsTracker.handleLaunchIntent(intent)`。
3. 从 intent 中读取：
   - `AppOpenFrom`
   - `Route`
   - `Scene`
   - `NoticeId`
   - `DistinctId`

上报规则：

| 来源字段 | 上报结果 |
| --- | --- |
| `AppOpenFrom=Push` | 公共属性 `traffic_source=fcm_push`，并上报 `notification_clicked`。 |
| `AppOpenFrom=app_push` | 公共属性 `traffic_source=app_push`。 |
| `AppOpenFrom=persistent` | 公共属性 `traffic_source=persistent`。 |
| `AppOpenFrom=app_shortcuts` | 公共属性 `traffic_source=app_shortcuts`。 |
| 无来源或其他来源 | 公共属性 `traffic_source=null`。 |

`Scene` 不为空时，宿主上报：

```kotlin
notification_app_clicked(scene)
```

这些是业务事件和公共事件属性，不属于用户属性白名单。

## 5. 常驻通知来源补充

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/service/notification/ToolNotificationIntentFactory.kt`

项目自己的常驻通知 PendingIntent 增加 SDK 兼容 extras：

```kotlin
putExtra("AppOpenFrom", "persistent")
putExtra("Route", route)
```

点击常驻通知进入 App 时，`MainActivity` 会识别为 `traffic_source=persistent`。

## 6. 首页和核心功能埋点

### 6.1 进入首页

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/app/AppRoot.kt`

监听当前导航 route：

1. route 属于 `AppRouteGroups.homeRoutes` 时，上报 `enter_homepage`。
2. 从引导页进入首页时，参数：`referrer_name=guidepage`。
3. 从核心功能返回首页时，参数：`referrer_name=corefeatures`。
4. 普通冷启动直接进入首页时，也会上报 `enter_homepage`，但不带 `referrer_name`。

### 6.2 点击核心功能

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/skin/home/HomeScreen.kt`

首页统一入口 `openFeature(feature)` 中调用：

```kotlin
AnalyticsTracker.trackCoreFeatureClicked(feature)
```

对应事件：

```text
click_corefeatures_btn(details)
```

### 6.3 进入核心功能页

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/app/AppRoot.kt`

监听 route 变化，通过 `AnalyticsTracker.featureForPrimaryRoute(route)` 判断是否为产品表定义的核心功能主页面，再调用：

```kotlin
AnalyticsTracker.trackCoreFeatureEntered(feature)
```

对应事件：

```text
enter_corefeatures_page(details)
```

### 6.4 `details` 映射

| 功能 | details |

| 设备信息 | `deviceInfo` |
| 电池信息 | `batteryInfo` |
| 应用使用情况 | `appUsage` |
| 通知栏清理 | `notificationClean` |

| 网络使用量查询 | `networkUsage` |
| 网络扫描 | `networkScan` |

进入功能页不再累计并上传 `total_battery_num`、`total_largefile_num` 等用户属性。

## 9. 权限弹窗埋点

### 9.1 通知权限弹窗

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/skin/common/components/NotificationPermissionPrompt.kt`

| 触发点 | 事件 | 参数 |
| --- | --- | --- |
| 通知权限弹窗关闭 | `notification_popup` | `noticeflag`, `if_ok` |
| 用户同意通知权限 | `agree_push_permission` | 无 |
| 用户不同意通知权限 | `disagree_push_permission` | 无 |

`noticeflag` 取值：

| 值 | 含义 |
| --- | --- |
| `1` | 启动页弹出 |
| `2` | 首页弹出 |
| `3` | 已使用过至少一次清理功能后，在首页弹出 |

权限结果事件只表达业务行为；`has_notification_permission` 用户属性由 SDK 刷新，宿主不写。

### 9.2 文件管理权限弹窗

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/skin/permission/PermissionAnalytics.kt`

### 9.3 权限流程结构化结果

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/core/permission/PermissionModels.kt`

`app/src/main/java/com/quickcleanpro/phonecleaner/use/core/permission/CleanXPermissionCoordinator.kt`

当前权限协调器对页面返回 `PermissionRequestResult`，页面可以按需处理，忽略返回值也兼容原有调用方式：

| 结果 | 含义 |
| --- | --- |
| `Started` | 已进入权限请求流程，可能展示自定义弹窗、系统权限弹窗或跳转设置页。 |
| `Granted` | 请求前或请求后已拿到权限。 |
| `Denied` | 用户拒绝或设置页返回后仍未授权。 |
| `Dismissed` | 用户关闭/取消自定义权限弹窗。 |
| `SettingsUnavailable` | 找不到可用设置页或权限处理不可用。 |
| `Busy` | 当前已有权限流程进行中，新请求不会静默丢弃。 |

权限流程 active 状态只用于广告运行态压制，不产生额外产品表事件。

## 12. 清理完成本地状态

代码位置：

`app/src/main/java/com/quickcleanpro/phonecleaner/use/core/common/operation/FeatureOperationTracker.kt`

`app/src/main/java/com/quickcleanpro/phonecleaner/use/core/analytics/AnalyticsTracker.kt`

原有功能操作统一经过 `DefaultFeatureOperationTracker.track(event)`，现在仍会调用：

```kotlin
AnalyticsTracker.trackFeatureOperation(event)
```

本地状态规则：

1. 清理类功能扫描完成且没有结果时，本地累计一次清理完成状态。
2. 清理类功能执行 `CLEAN`、`DELETE`、`REMOVE_LOCATION` 成功时，本地累计一次清理完成状态。
3. 该累计只用于 `AnalyticsTracker.hasCompletedCleanup()` 判断通知权限弹窗 `noticeflag=3`，不再上传 `total_cleancpl_num` 用户属性。

| 类型 | 代码位置 |

| 广告位 key 常量 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/ads/AdAreaKeys.kt` |

| 启动广告 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/ads/StartupAdCoordinator.kt` |
| 原生广告预加载 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/ads/AdvertisePreloader.kt` |
| 全屏广告运行态 gate | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/ads/AdPolicyGate.kt` |

宿主只传产品表里的 `areaKey`，例如：

```text
openPageAdv
foreGroundAdv
mainJunkCleanAdv
mainJunkCleanFinishAdv
returnFromJunkCleanAdv

```

广告 SDK 自动负责上报：

```text
ad_occur

ad_finish_loading
ad_impression
ad_click
ad_close
ad_revenue
```

宿主业务代码不得手动调用这些广告生命周期事件，避免重复上报。

## 14. App 自动采集说明

按 SDK 和产品口径，以下事件由 SDK 内部完成，宿主不补报：

```text
ta_app_install
ta_app_start
ta_app_view
ta_app_end
ta_app_crash
```

如果某个 SDK 版本内部实现和产品口径存在差异，也先以“不由宿主补报”为准，避免重复风险；需要调整时应优先在 SDK 层或产品口径层统一。

## 15. 推荐验证方式

按项目要求，不做不必要的完整编译、完整测试和完整打包。文档和代码更新后优先做以下轻量验证：

1. 静态检查宿主代码中不再上传禁用用户属性：`total_cleancpl_num`、`total_battery_num`、`total_largefile_num`、`ad_strategy`、`singular_`。
2. 静态检查宿主代码中不出现 SDK 自动事件手动上报：`ta_app_install`、`ta_app_start`、`ta_app_view`、`ta_app_end`、`ta_app_crash`。
3. 静态检查宿主代码中不出现广告生命周期事件手动上报：`ad_occur`、`ad_start_loading`、`ad_finish_loading`、`ad_impression`、`ad_click`、`ad_close`。
4. 静态检查 `MainActivity` 不再调用 `AnalyticsTracker.onAppForeground()` / `AnalyticsTracker.onAppBackground()`。
5. 静态检查 `MyApp` 通过 `ProcessLifecycleOwner` 维护使用时长。
6. 手动走冷启动、通知点击、首页进入功能、权限同意/拒绝、功能完成、功能返回首页主链路，确认业务事件仍可出现。

如需要做编译验证，建议只跑局部 Kotlin 编译或相关单测，并在实际执行后再补充结果。
