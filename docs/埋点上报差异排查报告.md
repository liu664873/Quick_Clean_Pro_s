## 2. 总体结论

### 2.1 需要重点确认或补齐的缺口

| 项 | 当前结论 | 影响 | 建议 |
| --- | --- | --- | --- |
| `ta_app_view` | Excel 要求存在；示例 SDK `initThinking()` 中 `APP_VIEW_SCREEN` 自动采集被注释；宿主未手动上报。 | 页面浏览自动采集可能缺失。 | 优先由 SDK 侧确认是否应开启 ThinkingData `APP_VIEW_SCREEN`；宿主不建议自行补，避免和 SDK 后续自动采集冲突。 |
| `ta_app_crash` | Excel 要求存在；示例 SDK `initThinking()` 中 `APP_CRASH` 自动采集被注释；宿主未手动上报。 | ThinkingData 崩溃自动采集可能缺失。 | SDK 侧确认是否开启 `APP_CRASH`，或明确由 Crashlytics/其他渠道承接。 |
| `adv_sdk_init` | Excel 要求事件名为 `adv_sdk_init`；SDK 实际上报 `adv_sdk_initcomplete`，且 `LogUtil` 会跳过 `adv_sdk_initcomplete` 的 ThinkingData 上报。 | 事件名和渠道口径不一致，后台查 `adv_sdk_init` 可能无数据。 | 产品/数据/SDK 统一事件名，或 SDK 增补/改名并确认是否需要进 ThinkingData。 |
| `notification_app_clicked` | 宿主只有 Intent 中 `Scene` 不为空才上报；常驻通知当前只带 `AppOpenFrom=persistent`、`Route`，不带 `Scene`。 | 常驻通知点击不会产生 `notification_app_clicked`，只会设置 `traffic_source=persistent`。 | 若常驻通知点击也要统计到该事件，需给常驻通知 extras 增加 `Scene` 或新增明确事件。 |
| `notification_clicked` 参数 | 宿主在 `AppOpenFrom=Push` 时上报，参数取 `NoticeId`、`DistinctId`；SDK 文档提示 FCM 本地通知构造中不一定写入这两个 extra。 | FCM 点击事件可能有事件但参数为空。 | SDK/服务端统一 FCM 点击 PendingIntent extras：`NoticeId`、`DistinctId`、`AppOpenFrom=Push`。 |

## 5. 广告和通知专项

### 5.1 广告生命周期边界

| 边界 | 当前结论 |
| --- | --- |
| 宿主职责 | 只传正确 `areaKey`，由 `AdAreaKeys.kt`、`AdPlacementRegistry.kt`、`InterstitialAdInterceptor.kt`、`StartupAdCoordinator.kt` 等完成场景到广告位映射。 |
| 宿主不做 | 不手动补报 `ad_occur`、`ad_start_loading`、`ad_finish_loading`、`ad_impression`、`ad_click`、`ad_close`。 |
| SDK 触发展示 | `ad_occur` 在全屏广告展示或从池中取广告时上报。 |
| SDK 加载 | `ad_start_loading` 在预加载和实时加载入口上报，`ad_finish_loading` 在加载成功回调上报。 |
| SDK 展示/收入 | `ad_impression` 与 `ad_revenue` 主要在 AdMob paid event、Native 展示回调、池复用展示链路上报。 |
| SDK 点击/关闭 | `ad_click` 在各广告类型点击回调上报，`ad_close` 在全屏广告关闭回调上报。 |

结论：广告相关 Excel 事件均不应由 QuickCleanPRO 宿主补报。若后台看不到广告事件，应优先检查 SDK 版本、广告位 key、广告网络回调和 ThinkingData/Firebase 分发规则。

| `app_shortcuts` | `app_shortcuts` |
| 其他或空 | `null` |

常驻通知由 `ToolNotificationIntentFactory` 写入 `AppOpenFrom=persistent`，因此点击常驻通知进入 App 时会设置 `traffic_source=persistent`。

## 6. 建议处理

| 优先级 | 项 | 建议 |
| --- | --- | --- |
| P0 | `ta_app_view` | 找 SDK/数据确认是否必须进入 ThinkingData。如果必须，建议 SDK 统一开启 `APP_VIEW_SCREEN` 自动采集，宿主不单独补。 |
| P0 | `ta_app_crash` | 找 SDK/数据确认是否必须进入 ThinkingData。如果必须，建议 SDK 统一开启 `APP_CRASH`，或明确由 Crashlytics 等平台替代并同步产品表。 |
| P0 | `adv_sdk_init` / `adv_sdk_initcomplete` | 统一 Excel 和 SDK 事件名；同时确认该初始化事件是否需要上报 ThinkingData。目前 SDK 会把 `adv_sdk_initcomplete` 排除在 ThinkingData 之外。 |
| P1 | `notification_app_clicked` 常驻通知缺口 | 如果产品希望统计常驻通知工具点击，给常驻通知 Intent 增加 `Scene`，或新增常驻通知点击事件，避免和 APP 场景化 Push 混淆。 |
| P1 | FCM 点击参数完整性 | 统一服务端 payload / SDK 本地通知 PendingIntent extras，确保点击到宿主时有 `AppOpenFrom=Push`、`NoticeId`、`DistinctId`。 |
| P2 | 用户属性更新方式 | `ab_test`、`network` 在 Excel 是 `setOnce`，SDK 当前是 `userSet`；`total_open_num`、`total_use_time` 在 Excel 是 `user_add`，宿主当前是本地累计后覆盖总值。建议数据侧确认是否接受当前累计总值口径。 |

## 7. 验证方式

按项目约束，本次不建议做不必要的完整编译、测试或打包。推荐验证如下：

### 7.1 静态验证

| 目标 | 建议命令/方式 |
| --- | --- |
| 确认宿主未手动补广告生命周期 | `rg -n "ad_occur|ad_start_loading|ad_finish_loading|ad_impression|ad_click|ad_close" app/src/main/java`，结果应只出现在文档或非手动上报说明中。 |
| 确认宿主未手动补 `ta_app_*` | `rg -n "ta_app_install|ta_app_start|ta_app_view|ta_app_end|ta_app_crash" app/src/main/java`，业务代码不应出现。 |
| 确认宿主业务事件入口 | `rg -n "trackGuide|trackHomeEntered|trackCoreFeature|trackNotification|trackFile|trackRate|trackPrivacy|trackTerms" app/src/main/java`。 |
| 确认 SDK 自动采集项 | 检查 `Ads.initThinking()` 中 `TDAnalytics.enableAutoTrack(...)`。 |
| 确认 SDK 广告事件 | 检查 `LogAdData.kt` 和各广告展示/加载类中的 `LogUtil.log(...)`。 |

### 7.2 必要手动验证

| 场景 | 预期 |
| --- | --- |
| 冷启动到首页 | `ta_app_start` 由 SDK 自动采集；宿主上报 `splash_display`、`enter_homepage`。 |
| 首次安装首次打开 | SDK 首次上报 `app_first_open`；ThinkingData 自动采集 `ta_app_install`。 |
| 引导页扫描、继续、跳过 | 对应 `enter_guide_page_scanning`、`enter_guide_page_scan_result`、`click_continue_guidepage`、`click_skip_guidepage`。 |
| 首页点击核心功能并进入页面 | 上报 `click_corefeatures_btn(details)`、`enter_corefeatures_page(details)`。 |
| 文件权限同意/拒绝 | 上报 `filemanager_popup(if_ok)` 和 `agree/disagree_filemanage_permission`。 |
| 通知权限同意/拒绝 | 上报 `notification_popup(noticeflag,if_ok)` 和 `agree/disagree_push_permission`。 |
| SDK 场景化通知展示 | SDK 上报 `notification_app_shown(scene)`，但超过 `log_time` 窗口可能跳过。 |
| FCM 点击启动 | 宿主设置 `traffic_source=fcm_push` 并上报 `notification_clicked`；参数取决于 `NoticeId`、`DistinctId` extras。 |
| 常驻通知点击启动 | 宿主设置 `traffic_source=persistent`；当前不应期待 `notification_app_clicked`。 |

## 8. 最终缺口清单

| 缺口/风险 | 是否需要 QuickCleanPRO 宿主立刻补报 | 原因 |
| --- | --- | --- |
| `ta_app_view` 未发现启用 | 否，先 SDK/数据确认 | 属于 ThinkingData 自动采集类事件，宿主补报容易和 SDK 后续自动采集重复。 |
| `ta_app_crash` 未发现启用 | 否，先 SDK/数据确认 | 属于 SDK/崩溃采集边界，宿主业务层不适合自行造事件。 |
| `adv_sdk_init` 名称不一致 | 否，先统一 SDK 和产品表 | SDK 当前实际事件名为 `adv_sdk_initcomplete`，需统一口径。 |
| 常驻通知不触发 `notification_app_clicked` | 视产品口径 | 如果产品只把 APP 场景化 Push 点击算作该事件，则无需处理；如果常驻通知点击也要统计，需要补 `Scene` 或新增事件。 |
| FCM 点击参数完整性依赖 payload/extras | 视服务端和 SDK 实现 | 宿主已经读取 `NoticeId`、`DistinctId`，但需要上游确保写入。 |
