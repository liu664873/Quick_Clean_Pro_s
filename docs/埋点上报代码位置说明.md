### 2.2 `noticeflag` 参数取值

`noticeflag` 由通知权限组件实时计算，代码位置：`app/src/main/java/com/quickcleanpro/phonecleaner/use/skin/common/components/NotificationPermissionPrompt.kt:166`。

| 值 | 含义 | 判断代码 |
| --- | --- | --- |
| `1` | 启动页弹出 | `isSplashVisible` 为 true |
| `2` | 首页常规弹出 | 不在启动页，且未完成过清理 |
| `3` | 使用过至少一次清理功能后，在首页弹出 | `AnalyticsTracker.hasCompletedCleanup()` 为 true |

清理完成状态只用于计算 `noticeflag=3`，不是 ThinkingData 事件。入口为 `FeatureOperationTracker.kt:47` 调用 `AnalyticsTracker.trackFeatureOperation(event)`，内部在 `AnalyticsTracker.kt:140` 根据清理成功/扫描无结果条件累加本地 `analytics_total_cleancpl_num`。

## 3. 公共事件属性 `traffic_source`

`traffic_source` 是公共事件属性，不是单独事件。宿主在 Activity 创建和收到新 Intent 时解析入口来源，然后通过 `AdvertiseSdk.setSuperProperties()` 写入。普通启动也会写 `traffic_source=null`，用于避免历史来源污染后续事件。

| 项目 | 说明 |
| --- | --- |
| 调用入口 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/app/MainActivity.kt:24`、`:40` |
| 解析位置 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/analytics/AnalyticsTracker.kt:49` |
| 写入位置 | `app/src/main/java/com/quickcleanpro/phonecleaner/use/core/analytics/AnalyticsTracker.kt:170` |
| SDK 方法 | `D:\pros\AdvertiseExample\advertise\src\main\java\com\pdffox\adv\AdvertiseSdk.kt:341` |
| Intent 字段 | `AppOpenFrom` |

| `AppOpenFrom` | `traffic_source` | 说明 |
| --- | --- | --- |
| `Push` | `fcm_push` | FCM 推送入口。 |
| `app_push` | `app_push` | App 场景化推送入口。 |
| `persistent` | `persistent` | 静默/常驻通知入口；宿主常驻通知在 `ToolNotificationIntentFactory.kt:51` 写入该值。 |

## 8. 静态结论

1. 宿主业务事件都集中在 `AnalyticsTracker`，通过 `AdvertiseSdk.logEvent()` 上报。
2. `traffic_source` 只作为公共事件属性写入，不触发通知点击事件。
3. 宿主用户属性已按前台 session 开始/结束上报，`total_open_num`、`total_use_time` 使用 `userAdd`，避免本地累计后覆盖总值。
4. SDK 内部负责的广告、通知展示、`ta_app_install`、`ta_app_start`、`ta_app_end`、`has_notification_permission`、`has_allfile_permission`、`ab_test`、`network`、`ip_info`，宿主不重复写。
5. `ta_app_view`、`ta_app_crash` 在示例 SDK 中仍是注释状态；如果数据侧必须要这两个自动事件，需要 SDK 版本确认或由 SDK 开启。
