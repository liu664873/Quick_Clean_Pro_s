# 埋点接入方案

## 1. 接入口径

本文档按最新口径整理宿主侧埋点接入范围：SDK 已经内部上报的广告、通知、`ta_app_*` 自动采集事件、SDK 内部用户属性，宿主侧不重复上报；宿主只负责业务漏斗、核心功能、权限弹窗、协议跳转，以及公共事件属性 `traffic_source`。

本次不要求修改业务代码、不要求编译、不要求打包。后续实现时建议继续保持一个统一埋点入口，例如现有 `AnalyticsTracker`，由页面或业务模块只调用语义化方法，避免到处直接调用 `AdvertiseSdk.logEvent`。

`app_first_open` 当前暂时不由宿主上报。`D:\pros\AdvertiseExample` 示例 SDK 中存在 `Ads.init()` 首次初始化上报 `app_first_open` 的逻辑；在未确认线上 SDK 行为前，宿主侧先不补报，避免重复。

## 2. 宿主事件清单

下表是宿主需要负责的事件范围。未出现在本表中的广告、通知 SDK 事件不纳入宿主待接入事件。

| 事件名 | 触发时机 | 参数 | 取值规则 | 建议埋点位置 |
| --- | --- | --- | --- | --- |
| `splash_display` | 开屏页关闭时上报 | `duration_time` | 数值，开屏页展示时长，建议毫秒口径保持一致 | 开屏页销毁、跳转或关闭前 |
| `enter_guide_page_scanning` | 进入引导页扫描中页面时上报 | 无 | 无 | 引导扫描中页面首次展示 |
| `enter_guide_page_scan_result` | 进入引导页扫描结果页面时上报 | 无 | 无 | 引导扫描结果页面首次展示 |
| `click_continue_guidepage` | 点击 `continue` / `get started` 按钮时上报 | 无 | 无 | 引导页继续按钮点击回调 |
| `click_skip_guidepage` | 点击 `skip` 按钮时上报 | 无 | 无 | 引导页跳过按钮点击回调 |
| `agree_filemanage_permission` | 用户同意文件管理权限时上报 | 无 | 无 | 文件管理权限授权结果回调 |
| `disagree_filemanage_permission` | 用户不同意文件管理权限时上报 | 无 | 无 | 文件管理权限拒绝或取消结果回调 |
| `agree_push_permission` | 用户同意通知权限时上报 | 无 | 无 | 通知权限授权结果回调 |
| `disagree_push_permission` | 用户不同意通知权限时上报 | 无 | 无 | 通知权限拒绝或取消结果回调 |
| `enter_homepage` | 进入首页或从功能页回到首页时上报 | `referrer_name` | `guidepage` 表示从引导页进入首页；`corefeatures` 表示从核心功能返回首页 | 首页路由展示或路由回退监听 |
| `click_corefeatures_btn` | 点击首页核心功能入口时上报 | `details` | 见下方核心功能取值表 | 首页功能按钮点击回调 |
| `enter_corefeatures_page` | 进入核心功能页面时上报 | `details` | 见下方核心功能取值表 | 核心功能页面路由进入或页面首次展示 |
| `rate_popup` | 评分弹窗点击任意按钮关闭时上报 | `if_ok` | 布尔值，按业务按钮结果传入 | 评分弹窗关闭回调 |
| `filemanager_popup` | 文件权限申请弹窗点击任意按钮关闭时上报 | `if_ok` | 布尔值，用户选择去设置/同意方向为 `true`，否则为 `false` | 文件权限申请弹窗关闭回调 |
| `notification_popup` | 通知权限弹窗关闭时上报 | `noticeflag`、`if_ok` | `noticeflag` 见下方规则；选择 setting 为 `true`，否则为 `false` | 通知权限弹窗关闭回调 |
| `check_privacy` | 跳转浏览器打开 Privacy Policy 时上报 | `referrer_name` | `launchpage`、`about` | 启动页或关于页隐私政策点击回调 |
| `check_terms` | 跳转浏览器打开 Terms of Service 时上报 | `referrer_name` | `launchpage`、`about` | 启动页或关于页服务协议点击回调 |

核心功能 `details` 取值固定如下：

| 功能 | `details` |
| --- | --- |
| 垃圾清理 | `junkClean` |
| 病毒扫描 | `virusAnti` |
| 应用锁 | `appLock` |
| 文件管理 | `fileManage` |
| 设备信息 | `deviceInfo` |
| 电池信息 | `batteryInfo` |
| 应用使用情况 | `appUsage` |

| WhatsApp 清理 | `whatsappClean` |
| 网络用量 | `networkUsage` |

`notification_popup.noticeflag` 取值规则固定如下：

| 场景 | `noticeflag` |
| --- | --- |
| 启动页弹出通知权限弹窗 | `1` |
| 首页弹出通知权限弹窗 | `2` |
| 至少使用过 1 次清理功能后，在首页弹出通知权限弹窗 | `3` |

## 3. 公共属性 `traffic_source`

`traffic_source` 是公共事件属性，不是单独事件。宿主在 App 启动或 `onNewIntent` 收到入口信息时设置，后续业务事件会自动带上该公共属性。

| 进入 App 的来源 | `traffic_source` |
| --- | --- |
| FCM 推送进入 App | `fcm_push` |
| 静默通知或常驻通知进入 App | `persistent` |
| App 发出的场景化推送进入 App | `app_push` |
| 快捷方式进入 App | `app_shortcuts` |
| 以上都不是，普通启动或无法识别来源 | `null` |

建议实现规则：

1. 在 `MainActivity.onCreate` 和 `MainActivity.onNewIntent` 统一解析启动 `Intent`。
2. 按 `AppOpenFrom` 或等价字段映射到上表值。
3. 通过 `AdvertiseSdk.setSuperProperties(mapOf("traffic_source" to value))` 写入公共属性。
4. 普通启动必须清空为 `null`，避免上一次推送来源污染后续普通启动事件。
5. 设置 `traffic_source` 时不额外上报 `notification_clicked` 或 `notification_app_clicked`。

## 4. SDK 已上报内容

以下内容由 SDK 内部负责，宿主侧不重复处理。

| 类型 | 已由 SDK 处理的内容 | 宿主策略 |
| --- | --- | --- |
| 广告生命周期事件 | `ad_occur`、`ad_start_loading`、`ad_finish_loading`、`ad_impression`、`ad_click`、`ad_close` 等广告相关事件 | 不补报，不在宿主业务事件中重复调用 |
| 通知 SDK 相关事件 | SDK 普通通知、场景化通知、FCM 通知展示/点击相关 SDK 事件 | 本方案不接 `notification_clicked`、`notification_app_clicked`；只设置 `traffic_source` |
| ThinkingData 自动采集 | `ta_app_install`、`ta_app_start`、`ta_app_end` | 由 SDK 初始化自动采集，宿主不补报 |
| SDK 用户属性 | `has_notification_permission`、`has_allfile_permission`、`ab_test`、`network`、`ip_info` | SDK 内部写入，宿主不重复 `setUserAttr` |

补充说明：`ta_app_view`、`ta_app_crash` 不属于本次宿主业务事件清单。若数据侧后续要求这两类基础自动采集，需要走 SDK 版本能力确认，而不是宿主在业务事件方案里自行补同名事件。

## 5. 特殊说明

### 5.1 `app_first_open` 暂不由宿主上报

`app_first_open` 虽然出现在原始事件表里，但当前最新口径是宿主暂时不报。原因是 SDK 示例代码中已经存在首次初始化时上报 `app_first_open` 的逻辑；宿主重复接入可能造成首次打开数翻倍。

后续如果确认线上 SDK 包没有上报 `app_first_open`，再单独补宿主方案：在首次启动完成 SDK 初始化后，使用本地持久化标记做 once guard，只上报一次 `app_first_open`。

### 5.2 通知点击事件不进宿主事件清单

`notification_clicked` 和 `notification_app_clicked` 当前不写入宿主待接入事件。通知入口只负责设置公共属性 `traffic_source`。如果后续数据口径明确要求宿主统计通知点击，再单独补充通知点击事件的参数、extra 来源和去重规则。

### 5.3 用户属性边界

本方案不要求宿主补写 SDK 内部用户属性：`has_notification_permission`、`has_allfile_permission`、`ab_test`、`network`、`ip_info`。权限同意/拒绝事件只表达用户行为，不代表宿主需要维护对应权限用户属性。

## 6. 验证建议

本方案避免不必要的编译、测试和打包。接入完成后的验证以静态检查和少量手动验证为主。

静态检查建议：

1. 确认宿主业务事件只包含本文档第 2 节表格中的事件。
2. 确认宿主没有新增广告生命周期事件补报。
3. 确认宿主没有新增 `notification_clicked`、`notification_app_clicked` 补报。
4. 确认 `traffic_source` 覆盖 `fcm_push`、`persistent`、`app_push`、`app_shortcuts`、`null` 五类结果。
5. 确认宿主没有重复写 SDK 用户属性：`has_notification_permission`、`has_allfile_permission`、`ab_test`、`network`、`ip_info`。

手动验证建议：

| 场景 | 期望结果 |
| --- | --- |
| 普通启动并关闭开屏页 | 上报 `splash_display`，公共属性 `traffic_source=null` |
| 引导页完整流程 | 依次触发引导页展示、继续/跳过相关事件 |
| 从引导页进入首页 | 上报 `enter_homepage(referrer_name=guidepage)` |
| 点击并进入核心功能 | 点击时报 `click_corefeatures_btn(details=...)`，进入时报 `enter_corefeatures_page(details=...)` |
| 从核心功能返回首页 | 上报 `enter_homepage(referrer_name=corefeatures)` |
| 权限弹窗同意/拒绝 | 上报对应同意/拒绝事件，SDK 权限用户属性不由宿主重复写 |
| FCM 推送启动 | 设置 `traffic_source=fcm_push`，不额外上报通知点击事件 |
| 常驻通知或静默通知启动 | 设置 `traffic_source=persistent`，不额外上报通知点击事件 |
| 场景化 App 推送启动 | 设置 `traffic_source=app_push`，不额外上报通知点击事件 |
| 快捷方式启动 | 设置 `traffic_source=app_shortcuts` |
